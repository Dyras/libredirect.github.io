Why forking Privacy Redirect?

Privacy Redirect hasn't been maintained since 6/12/2021. Many instances are hard coded and should be removed as they became offline and many others should be added. Many Pull Requests were just pending and waiting. So we stepped in. This was the main reason. Later though, we started improving more by adding new features and frontends.

Why all those permissions?

let's go on them one by one:

  • webRequest, webRequestBlocking: To block http requests or redirect them. Examply like, To redirect to Invidious before it can reach YouTube's servers.
  • storage: To save LibRedirect's settings permanently.
  • cookies: To apply settings across multiple instances. Note that we also use localStorage. Those options are disabled by default though.
  • menus: For adding some buttons to the right click menu:
  • <all_urls>: LibRedirect is dynamic and customizable. The targets e.g youtube.com, twitter.com are written in Regex and are much more complicated than to be hard-coded in the manifest. Ex: search Regex. Further more, we need to access the instances sites too to inject some localStorage variables.

Where the hell are those instances coming from?

Every frontend project has a list of public instances. Ex: Invidious' official instances list, Nitter's official instances list. We fetch those lists using an automated python script. This script gets triggered every 3 days. We don't intercept it nor modify the lists. They're not our responsibility, they're the projects maintainers' except Piped because the Project Maintainer doens't maintain URL of Instances rather API links to connect to other Instances via Frontend.

Why not using a single reputable instance?

We can make all the redirections based on e.g pussthecat.org. This server is a really reputable and trusted server. Problem is that we'll just kill it with DDoS attacks. Most of these servers are small and some are even hosted from homes. Though if they were somehow able to handle the load, then they won't be able to handle the costs.

At the end, each person should pick and use one or more instances that the user trusts. It's their own choice.

It's not working at all!

This happens weirdly after updating to a new version. Just reinstall it. If it didn't help, open an issue.

Chrome Web Store?

We can't publish it to the Chrome Web Store as it requires manifest v3 and we use manifest v2. You can still Install it though. We are planning to adapt to Manifest v3 in Firefox as they will support the feature Libredirect needs. If other Browsers do support Mozilla Policy. You might see Libredirect in Chrome Web Store.

Why not manifest v3?

The Problem

As what @SimonBrazell (Author of Privacy Redirect from which Libredirect is forked) said here: https://github.com/SimonBrazell/privacy-redirect/pull/319#issuecomment-985895209

"Redirecting and blocking urls is now so impossible to do with the webRequest function. If you take look at the manifest.json files of Privacy Redirect, uBlock Origin, and Redirector, you'll see that they all use manifest v2. I can't add nor publish a manifest v2 to the chrome web store. Read the warning in this page: https://developer.chrome.com/docs/extensions/mv2/

Alternatives to WebRequest ?

Further reading and I saw an alternative that is declarativeNetRequest. It's syntax which is a static one and is like a settings page for chrome to use rather than with webrequest where you had to pass the request to the extension and wait for it's JavaScript code to process to respond. ex: Imgur Redirect Function. There's a whole lot of difference between static settings and a whole language like JavaScript processing and understanding your input and giving you an output.

What to do?

The only solution is to stay on manifest v2 rather than manifest v3. The extension will still work on chromium browsers, but it can't be in the chrome web store.

Thankfully there are other extension stores for chromium browsers like Edge Add-On Store, Opera Addons

Edge Add-Ons doesn't accept installing an extension from their store if you're not using Edge. Opera add-ons is the same story.

What is Unify settings in the Popup ?

This setting is to make the use of multiple instances consistent by unifying all the settings such as theme, language, etc...

You use it by visiting one of your selected instances. Go to it's settings page. Change it's settings.

Then press the Unify Button from Popup. This will copy all the settings from the current instance to all the other selected instances.

Usually they're stored using cookies. We just copy those cookies to the other instances. By The Way, We don't copy critical cookies that have tokens or credential data. We copy only the cookies for customization. Some instances use localStorage instead of cookies such as Piped. We copy it too but with a different approach, though you as the user will have the same result.

Removed Functionalities and Changes

As of Libredirect v2.0, The Following Functionalities are removed or replaced :

  • Custom Settings for Frontends, instead Unify Settings will be used.
  • Removed 'Bypass feature from YouTube, Nitter and Libreddit', If you want to reinstate back in the next release. Raise your voice here in this Github Issue : https://github.com/libredirect/libredirect/issues/277
  • Always used Preferred Instances because it just makes more confusion and complexity to the code than it's benefits.
  • Instances that use Cloudflare CDN are disabled by default because Cloudflare CDN may acts as an Man-in-the-Middle Attack and get unencrypted data about you.
  • Frontends such as Librespeed, PixivMoe and Soju are removed.
  • Beatbump is disabled by Default because the Only Instance of it uses Clouflare CDN.
